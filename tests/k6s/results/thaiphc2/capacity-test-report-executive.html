รายงานผลการทดสอบความพร้อมรองรับผู้ใช้งาน

ระบบสารสนเทศสาธารณสุขชุมชน (Thai PHC HSS API) - ระบบตัวกลางรองรับ อสม. 1.08 ล้านคนทั่วประเทศ
วันที่ทดสอบ: 7 มีนาคม 2569 สภาพแวดล้อม: Kubernetes Cluster (Production-equivalent) เครื่องมือ: Grafana K6 (มาตรฐานสากล)
1. บทสรุปผู้บริหาร (Executive Summary)

ระบบ Thai PHC HSS API เป็นระบบตัวกลาง (Central Platform) ที่ให้บริการข้อมูลสาธารณสุขชุมชนแก่ผู้ใช้งาน อสม. ทั่วประเทศ จำนวน 1,080,000 คน รวมถึงทำหน้าที่เป็น ระบบยืนยันตัวตนกลาง (Single Sign-On / SSO) ให้กับระบบย่อยอื่นๆ เช่น Smart OSM API และ E-Learning API

การทดสอบครั้งนี้มีวัตถุประสงค์เพื่อยืนยันว่าระบบสามารถรองรับผู้ใช้งานจริง ในช่วงที่มีการใช้งานสูงสุด (เช่น ช่วงวันที่ 25-สิ้นเดือนของทุกเดือน) โดยจำลองผู้ใช้งานพร้อมกัน 3 ระดับ: 5,000 / 10,000 / 20,000 คน

2. วิธีการทดสอบ
จำลองการใช้งานจริงของ อสม. ภายในระบบ Kubernetes Cluster

การทดสอบใช้เครื่องมือ Grafana K6 ซึ่งเป็นเครื่องมือทดสอบประสิทธิภาพมาตรฐานสากล โดยจำลองสถานการณ์การใช้งานจริงของ อสม. 1 คน ดังนี้:

ขั้นตอน รายละเอียด ความหมาย
1. ตรวจสอบสถานะระบบHealth Checkระบบพร้อมใช้งานหรือไม่
2. เข้าสู่ระบบ / ยืนยันตัวตนAuth / SSOล็อกอินและตรวจสอบสิทธิ์
3. ดูข้อมูลส่วนตัวโปรไฟล์ อสม.เปิดหน้าข้อมูลตนเอง
4. ดูข้อมูลพื้นที่จังหวัด/อำเภอ/ตำบลเลือกพื้นที่รับผิดชอบ
5. ดูประกาศข่าวสาร/ประกาศอ่านประกาศล่าสุด
6. ดูหน้าสรุป Dashboardสรุปข้อมูลภาพรวมหน้าจอหลักของระบบ

แต่ละรอบการใช้งาน มีช่วงพัก (Think Time) 3-7 วินาที เพื่อจำลองพฤติกรรมจริง ที่ผู้ใช้อ่านข้อมูลบนหน้าจอก่อนกดหน้าถัดไป

การทดสอบดำเนินการภายใน Kubernetes Cluster โดยตรง จึงไม่มีความล่าช้าจากเครือข่ายอินเทอร์เน็ตมาเป็นตัวแปร ทำให้ผลลัพธ์สะท้อนความสามารถที่แท้จริงของระบบ

4. การประเมินจำนวนผู้ใช้งานที่รองรับได้
พิจารณาจากพฤติกรรมการใช้งานจริงของ อสม.

ในการใช้งานจริง ผู้ใช้ 1 คน ไม่ได้ส่งคำขอตลอดเวลา แต่จะมีช่วงอ่านข้อมูล (Think Time) ระหว่างการกดแต่ละครั้ง ดังนั้นจำนวนผู้ใช้ที่ระบบรองรับได้จริง ขึ้นอยู่กับความถี่ในการใช้งานของผู้ใช้แต่ละคน

4.1 ขีดความสามารถสูงสุดของระบบ

ระบบสามารถประมวลผลได้สูงสุดประมาณ 6,000 คำขอต่อวินาที โดยยังคงอัตราความสำเร็จเกิน 99% (ทดสอบจริงที่ 5,918 คำขอ/วินาที ด้วย 10,000 ผู้ใช้พร้อมกัน ได้ 99.40%)

4.2 ประมาณจำนวนผู้ใช้งานจริงที่รองรับได้

รูปแบบการใช้งาน ตัวอย่างพฤติกรรม คำขอ/วินาที/คน รองรับพร้อมกันได้
ใช้งานเบา เปิดอ่านประกาศ เช็คข้อมูล 1 ~6,000 คน
ใช้งานปกติ กดเมนู กรอกข้อมูล ดู Dashboard 3 ~2,000 คน
ใช้งานหนัก ค้นหา โหลดรายงาน กรอกแบบฟอร์ม 5 ~1,200 คน
ใช้งานหนักมาก เปิดหลายหน้าจอ รีเฟรชบ่อย 10 ~600 คน

5. ความสามารถในการเป็นระบบตัวกลาง (SSO / Central Platform)
ประเมินการรองรับเมื่อระบบอื่นเชื่อมต่อเข้ามาใช้บริการ

ระบบ Thai PHC HSS API ทำหน้าที่เป็น ระบบตัวกลาง 2 บทบาท:

  1. ศูนย์ยืนยันตัวตน (SSO - Single Sign-On): ระบบย่อยอื่นๆ เช่น Smart OSM, E-Learning ใช้ API เข้าสู่ระบบ และตรวจสอบสิทธิ์ผ่านระบบนี้
  2. ศูนย์ข้อมูลกลาง: ให้บริการข้อมูลพื้นฐาน เช่น ข้อมูล อสม., ข้อมูลพื้นที่, ข้อมูลจังหวัด/อำเภอ/ตำบล แก่ระบบย่อย

5.1 การใช้ทรัพยากรเมื่อเป็นตัวกลาง

ระบบที่เชื่อมต่อ ฟังก์ชันที่ใช้ คำขอเพิ่ม/ผู้ใช้ หมายเหตุ
Smart OSM API SSO Login + ข้อมูล อสม. ~2-3 คำขอ/session เฉพาะตอนเข้าสู่ระบบ
E-Learning API SSO Login ~1-2 คำขอ/session เฉพาะตอนเข้าสู่ระบบ
ระบบอื่นในอนาคต SSO Login + ข้อมูลพื้นฐาน ~2-5 คำขอ/session ขึ้นอยู่กับการออกแบบ

การใช้งาน SSO จะเพิ่มภาระเพียงเล็กน้อย เนื่องจากคำขอ SSO เกิดขึ้นเฉพาะตอนเข้าสู่ระบบ (ครั้งเดียวต่อ session ไม่ใช่ทุกครั้งที่กดหน้าจอ) เมื่อเข้าสู่ระบบแล้ว ระบบย่อยจะทำงานด้วย Token ของตัวเอง โดยไม่ต้องถามระบบตัวกลางซ้ำ

5.2 ประเมินภาระรวมเมื่อเปิดใช้ทุกระบบพร้อมกัน

สถานการณ์ คำขอ/วินาที ที่ต้องการ ขีดจำกัดระบบ ผลประเมิน
Thai PHC อย่างเดียว (ปกติ) ~1,350-2,700 ~6,000 รองรับได้สบาย
Thai PHC + SSO ระบบย่อย 2 ระบบ ~1,800-3,500 ~6,000 รองรับได้
Thai PHC + SSO ระบบย่อย 5 ระบบ ~2,500-5,000 ~6,000 ใกล้ขีดจำกัด
ทุกระบบ + กรณีเร่งด่วน (Peak) ~5,000-8,000 ~6,000 อาจเกินขีดจำกัด
ข้อสังเกต: หากในอนาคตมีระบบย่อยเชื่อมต่อเพิ่มมากกว่า 5 ระบบ หรือเกิดสถานการณ์ที่ อสม. จำนวนมากเข้าใช้งานพร้อมกันอย่างกะทันหัน ควรพิจารณาเพิ่มทรัพยากรเซิร์ฟเวอร์ (เพิ่ม Pods) ซึ่งระบบออกแบบมาให้สามารถขยายได้อัตโนมัติ (Auto-Scaling)

7. สรุปผลและข้อเสนอแนะ

ผลการทดสอบยืนยันว่าระบบ Thai PHC HSS API พร้อมรองรับ อสม. 1.08 ล้านคนทั่วประเทศ ในสถานการณ์การใช้งานปกติ โดยมีรายละเอียดดังนี้:

หัวข้อ ผลประเมิน
รองรับผู้ใช้ อสม. 1.08 ล้านคน (ทยอยใช้ 5 วัน) รองรับได้ - เหลือ headroom 2-4 เท่า
รองรับผู้ใช้พร้อมกัน 5,000 คน สำเร็จ 99.94%
รองรับผู้ใช้พร้อมกัน 10,000 คน สำเร็จ 99.40%
เป็นตัวกลาง SSO ให้ระบบย่อย 2-3 ระบบ รองรับได้
เป็นตัวกลาง SSO ให้ระบบย่อย 5+ ระบบ ต้องเพิ่มทรัพยากร
ขีดจำกัดสูงสุดของระบบปัจจุบัน ~6,000 คำขอ/วินาที (~10,000 ผู้ใช้พร้อมกัน)

ข้อเสนอแนะ:

  1. สำหรับการใช้งานปัจจุบัน: ระบบพร้อมใช้งานได้ทันที โดยไม่จำเป็นต้องเพิ่มทรัพยากรเพิ่มเติม
  2. หากต้องการรองรับมากขึ้น: สามารถขยายจำนวนเซิร์ฟเวอร์ (Pods) จาก 35 เป็น 50-70 ตัว เพื่อรองรับผู้ใช้พร้อมกัน 15,000-20,000 คน ซึ่งเป็นการปรับค่าระบบเท่านั้น ไม่ต้องแก้ไขโปรแกรม
  3. การเชื่อมต่อระบบย่อยใหม่: สามารถเพิ่มระบบย่อยที่ใช้ SSO ได้อีก 2-3 ระบบ โดยไม่กระทบประสิทธิภาพ หากเกินกว่านั้นควรเพิ่มทรัพยากร
