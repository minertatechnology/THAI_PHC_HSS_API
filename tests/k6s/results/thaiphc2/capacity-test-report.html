ทดสอบรองรับผู้ใช้งานพร้อมกัน 5,000 / 10,000 / 20,000 คน สำหรับระบบ อสม. 1.08 ล้านคน
วันที่ทดสอบ: 7 มีนาคม 2569 เวอร์ชัน: v3.124 (Early Cache Check) สภาพแวดล้อม: Kubernetes Cluster (In-Cluster) Pods: 35 | Namespace: aorsormor-system เครื่องมือ: Grafana K6
สรุปผลการทดสอบ Capacity Test (Executive Summary)

1. วัตถุประสงค์: ทดสอบความสามารถสูงสุดในการรองรับผู้ใช้งานพร้อมกัน (Capacity Test) ของระบบ Thai PHC HSS API (ระบบสารสนเทศสาธารณสุขชุมชน) ซึ่งรองรับ อสม. ทั่วประเทศ 1.08 ล้านคน โดยจำลองสถานการณ์ 3 ระดับ: 5,000 VUs (ปกติ), 10,000 VUs (สูง), 20,000 VUs (เกินขีดจำกัด) ภายใน Kubernetes Cluster เพื่อให้ผลลัพธ์แม่นยำโดยไม่มีคอขวดจากเครือข่ายภายนอก

2. สถานการณ์ทดสอบ (User Journey): จำลองการใช้งานจริงของ อสม. 1 คน ประกอบด้วย 6 ขั้นตอน: (1) Health Check (2) ตรวจสอบตัวตน (Auth/Me) (3) ดูข้อมูลส่วนตัว (OSM/Me) (4) ดูรายการจังหวัด (Lookups) (5) ดูประกาศ (Announcements) (6) ดู Dashboard สรุปข้อมูล แต่ละรอบใช้เวลา Think Time 3-7 วินาที เพื่อจำลองพฤติกรรมผู้ใช้จริง

ข้อเสนอแนะ (Recommendations)

1. คอขวดที่เหลือ (ความเร่งด่วน: ต่ำ-ปานกลาง)

  1. Health Check Endpoint: p(95) เกินเกณฑ์ 2 วินาทีทุกระดับ สาเหตุ: health check ทำ DB ping ซึ่งช้าลงเมื่อ load สูง แนะนำ: ใช้ in-memory check หรือ cache ผล DB ping ไว้ 5-10 วินาที
  2. Auth/Me Endpoint: p(95) เกินเกณฑ์ 3 วินาทีที่ 10K+ VUs สาเหตุ: JWT verification + DB lookup ทุก request แนะนำ: เพิ่ม TTL ของ session cache จาก 5 นาทีเป็น 10-15 นาที

2. หากต้องรองรับ 20K+ VUs ในอนาคต

  1. เพิ่ม Pods: จาก 35 เป็น 50-70 pods เพื่อกระจาย connection load
  2. เพิ่ม PgBouncer pool: เพิ่ม max connections ใน PgBouncer ให้รองรับ pods ที่เพิ่มขึ้น
  3. Nginx tuning: เพิ่ม worker_connections, proxy_connect_timeout
  4. Database Read Replica: แยก read traffic ไป replica เพื่อลดภาระ primary

3. ข้อเสนอแนะทั่วไป

  1. Soak Test: ทดสอบ 2,000-3,000 VUs เป็นเวลา 1-2 ชั่วโมง เพื่อตรวจสอบ Memory Leak
  2. Monitoring: ติดตั้ง Grafana + Prometheus เพื่อ Real-time monitoring
สรุป (Conclusion)

ระบบ Thai PHC HSS API v3.124 พร้อมรองรับ อสม. 1.08 ล้านคนทั่วประเทศ ในสถานการณ์ปกติ

ผลการทดสอบ Capacity Test 3 ระดับ ยืนยันว่า:

  • 5,000 VUs (ปกติ): 99.94% success | 3,231 req/s — ผ่านทุกเกณฑ์
  • 10,000 VUs (สูง): 99.40% success | 5,918 req/s — ผ่านเกือบทุกเกณฑ์
  • 20,000 VUs (เกิน): 52.51% success | 8,259 req/s — เกินขีดจำกัดของระบบ
  • Breaking Point: ~15,000 VUs (~6,000 req/s)

สำหรับการใช้งานจริง (อสม. ทยอยใช้ 5 วัน, think time 10-60 วินาที) ระบบรองรับได้ เกินกว่าความต้องการจริง 2-5 เท่า หากเกิดสถานการณ์ที่ผู้ใช้จำนวนมากเข้าใช้พร้อมกัน (flash crowd) สามารถ scale pods เพิ่มเติมได้ตาม HPA configuration
